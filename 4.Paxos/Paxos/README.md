# Single-decree Paxos

### Paxos

Paxos — это классический алгоритм консенсуса, который позволяет группе узлов договориться о единственном значении даже при наличии сбоев и задержек в сети. Алгоритм гарантирует, что:
- Будет выбрано не более одного значения (safety)
- В конечном итоге значение будет выбрано (liveness при определённых условиях)
- Выбранное значение должно быть одним из предложенных

### Архитектура системы

Система состоит из двух типов узлов:

- **Acceptor** — узлы, которые принимают участие в голосовании и хранят принятые значения
- **Proposer** — узлы, которые предлагают значения и координируют процесс достижения консенсуса

### Алгоритм Paxos

Paxos работает в две фазы:

#### Phase 1: Prepare (Подготовка)
1. **Proposer** выбирает уникальный номер предложения (proposal number) `n` и отправляет сообщение `PREPARE(n)` всем Acceptor-ам
2. **Acceptor** получает `PREPARE(n)`:
   - Если `n` больше любого ранее полученного номера, Acceptor обещает не принимать предложения с меньшими номерами
   - Acceptor отвечает `PROMISE(n, accepted_value, accepted_number)`, где `accepted_value` и `accepted_number` — ранее принятое значение (если есть)
3. **Proposer** ждёт ответов от большинства (quorum) Acceptor-ов

#### Phase 2: Accept (Принятие)
1. **Proposer** получив ответы от большинства:
   - Если хотя бы один Acceptor уже принял значение, Proposer должен использовать значение с наибольшим `accepted_number`
   - Иначе Proposer может использовать своё изначальное значение
2. **Proposer** отправляет `ACCEPT(n, value)` всем Acceptor-ам
3. **Acceptor** получает `ACCEPT(n, value)`:
   - Если не было обещаний с большим номером, принимает значение
   - Отвечает `ACCEPTED(n, value)`
4. **Proposer** получив подтверждения от большинства, считает значение принятым

### Протокол сообщений

#### Локальные сообщения (от клиента к Proposer):
- `PROPOSE_REQUEST` с параметром `value` — предложить значение для консенсуса

#### Ответы от Proposer (локальные):
- `PROPOSE_RESPONSE` с параметром `value` — значение, по которому достигнут консенсус

#### Между Proposer и Acceptor:

**Phase 1:**
- `PREPARE` с параметром `proposal_number` — запрос на подготовку
- `PROMISE` с параметрами `proposal_number`, `accepted_number` (может быть null), `accepted_value` (может быть null) — обещание

**Phase 2:**
- `ACCEPT` с параметрами `proposal_number`, `value` — запрос на принятие значения
- `ACCEPTED` с параметрами `proposal_number`, `value` — подтверждение принятия

### Требования к реализации

1. **Уникальность номеров предложений**: Каждый Proposer должен генерировать уникальные proposal numbers (можно использовать комбинацию счётчика и ID процесса)
2. **Работа с большинством**: Операции завершаются при получении ответов от большинства (quorum) Acceptor-ов
3. **Корректность при конфликтах**: Если несколько Proposer-ов одновременно предлагают значения, система должна выбрать одно
4. **Устойчивость к сбоям**: Алгоритм должен работать даже если меньшинство узлов недоступно
5. **Retry механизм**: Если Proposer не смог достичь консенсуса (например, другой Proposer использовал больший номер), он должен повторить попытку с новым, большим номером

### Важные замечания

1. Для простоты можно считать, что сеть в конечном итоге доступна, но могут быть временные разделения
2. Если Proposer не смог достичь консенсуса (получил отказы или не набрал quorum), он должен начать всё заново с большим proposal number
3. Acceptor-ы не общаются друг с другом напрямую
4. Количество Acceptor-ов фиксировано и передаётся Proposer при инициализации

## Реализация

Для реализации и тестирования решения используется учебный фреймворк anysystem. В папке задачи размещена заготовка для решения `solution.py`. Вам необходимо доработать реализацию классов `Acceptor` и `Proposer` так, чтобы проходили все тесты.

При инициализации:
- **Proposer** получает свой уникальный ID и список ID всех Acceptor-ов в системе

Количество Acceptor-ов во время теста не меняется.

## Запуск тестов

Перед запуском тестов убедитесь, что на вашей машине [установлен Rust](https://www.rust-lang.org/tools/install).

### Основные команды

```bash
cd tests
# Запуск всех тестов с вашим решением
cargo run -- -i ../solution.py

# Запуск конкретного теста
cargo run -- -i ../solution.py -t "SINGLE WRITE-READ"

# Запуск с отладочной информацией  
cargo run -- -i ../solution.py -d
```

Если вы найдете ошибки или требования из условий, которые не покрывают наши тесты, то вы можете получить за это бонусы. Готовые тесты оформляйте как pull request в репозиторий курса.

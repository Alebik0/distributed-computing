# Репликация атомарного регистра (ABD алгоритм)

## Описание задачи

В этой задаче вам необходимо реализовать **атомарный регистр** с репликацией, используя классический **ABD алгоритм** (Attiya, Bar-Noy, Dolev).

### Что такое атомарный регистр?

Атомарный регистр — это распределённая структура данных, которая хранит одно значение и поддерживает две операции:
- **READ** — чтение текущего значения
- **WRITE** — запись нового значения

**Атомарность** означает, что все операции выглядят так, как будто они выполняются мгновенно в некотором глобальном порядке, даже если на самом деле они выполняются параллельно на разных узлах.

### ABD алгоритм

ABD алгоритм обеспечивает атомарность в системе с репликами без использования консенсуса. Каждое значение регистра имеет **timestamp** (метку времени), которая помогает определить порядок операций.

#### Фазы алгоритма:

**WRITE(value):**
1. **READ phase**: Клиент читает текущий timestamp со всех реплик (посылает READ_TS запросы)
2. Ждёт ответов от большинства (majority) реплик
3. Выбирает максимальный timestamp и увеличивает его на 1
4. **WRITE phase**: Клиент отправляет новое значение с новым timestamp на все реплики
5. Ждёт подтверждений от большинства реплик
6. Возвращает успех

**READ():**
1. **READ phase**: Клиент читает пары (timestamp, value) со всех реплик
2. Ждёт ответов от большинства реплик
3. Выбирает значение с максимальным timestamp
4. **WRITE-BACK phase**: Клиент отправляет это значение обратно на все реплики (для распространения)
5. Ждёт подтверждений от большинства реплик
6. Возвращает прочитанное значение

#### Важные свойства:

- **Quorum** — операции требуют ответа от большинства реплик (majority)
- **Timestamp** — монотонно возрастающий счётчик, определяющий порядок операций
- **Write-back в READ** — обеспечивает, что новые операции увидят последнее записанное значение

### Формат сообщений

#### Локальные сообщения (от теста к клиенту):
- `GET` — прочитать значение регистра
- `PUT` с параметром `value` — записать значение в регистр

#### Ответы от клиента (локальные):
- `GET_RESP` с параметром `value` — ответ на чтение
- `PUT_RESP` — подтверждение записи

#### От клиента к регистру:
Вам нужно реализовать протокол обмена сообщениями между клиентом и репликами. Например:
- `READ_TS` — запрос текущего timestamp и значения
- `WRITE_VAL` с параметрами `ts`, `value` — запись значения с timestamp

#### От регистра к клиенту:
- `READ_TS_RESP` с параметрами `ts`, `value` — ответ с timestamp и значением
- `WRITE_ACK` — подтверждение записи

### Архитектура

В системе есть два типа процессов:

- **Регистры (Registers)**: Реплики, которые хранят копии данных
  - Каждая реплика хранит текущее значение и timestamp
  - Реплики отвечают на запросы от клиентов
  - Реплики не общаются напрямую друг с другом
  
- **Клиенты (Clients)**: Процессы, которые выполняют операции чтения/записи
  - Клиент знает список всех реплик
  - Клиент координирует двухфазный протокол ABD
  - Клиент ждёт ответов от большинства (quorum) реплик
  - Может быть несколько клиентов (писателей и читателей) одновременно

### Требования

1. **Атомарность**: Операции READ и WRITE должны быть атомарными (линеаризуемыми)
2. **Работа с большинством**: Операции завершаются при получении ответов от большинства реплик
3. **Корректность при сбоях**: Система должна работать корректно даже если меньшинство узлов недоступно
4. **Множественные клиенты**: Поддержка одновременных операций от разных клиентов

Вам нужно реализовать два класса `ABDRegister` и `ABDClient` в файле `solution.py`.

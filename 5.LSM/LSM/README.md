# LSM-дерево (Log-Structured Merge-Tree)

## Описание задания

В этом задании вам предстоит реализовать полнофункциональную структуру данных LSM-дерево (Log-Structured Merge-Tree), которая используется в современных NoSQL базах данных, таких как Cassandra, RocksDB, LevelDB и других.

LSM-дерево — это структура данных, оптимизированная для операций записи. Она позволяет эффективно обрабатывать большие объемы данных путем буферизации записей в памяти и периодического сброса их на диск в виде отсортированных файлов с последующим их слиянием (compaction).

Язык программирование - ваш выбор!

## Требования к реализации

### 1. SSTable (Sorted String Table)

Реализуйте структуру SSTable для хранения отсортированных данных на диске:

- **Memory-mapped файлы**: используйте `mmap` для эффективного доступа к данным на диске без явных операций чтения
- **Блокировка**: обеспечьте потокобезопасность при одновременном доступе к SSTable из нескольких потоков
- **Формат хранения**: 
  - Ключи и значения должны быть отсортированы по ключу
  - Поддержка эффективного поиска по ключу
  - Метаданные файла (индексы, статистика)

### 2. MemTable

Реализуйте in-memory буфер для операций записи:

- Используйте сбалансированную структуру данных (например, Skip List)
- Поддержка операций `put(key, value)`, `get(key)`, `delete(key)`
- Автоматический flush на диск при достижении порогового размера

### 3. Write-Ahead Log (WAL)

Обеспечьте надежность данных через журнал опережающей записи:

- **WAL файл**: все операции записи сначала фиксируются в WAL
- **Recovery**: восстановление состояния MemTable после сбоя из WAL
- **Ротация**: создание нового WAL после flush MemTable на диск
- **Очистка**: удаление устаревших WAL файлов

### 4. Tombstones и Garbage Collection

Реализуйте механизм удаления записей:

- **Tombstones**: вместо физического удаления записывайте специальные маркеры удаления
- **Garbage Collection**: удаление tombstones и устаревших версий записей во время compaction
- Учет времени жизни tombstones для корректной работы с конкурентными операциями

### 5. Multi-way Compaction (K-way Merge)

Реализуйте процесс слияния нескольких SSTable файлов:

- **K-way merge**: одновременное слияние K SSTable файлов в один
- **Приоритетная очередь**: использование min-heap для эффективного слияния
- **Устранение дубликатов**: сохранение только последней версии каждого ключа
- **Удаление tombstones**: очистка удаленных записей
- **Уровневая организация**: поддержка нескольких уровней SSTable (Level 0, Level 1, и т.д.)

### 6. Adaptive Compaction Policy

Реализуйте адаптивную стратегию compaction:

- **Size-tiered compaction**: объединение файлов схожего размера
- **Leveled compaction**: поддержка уровневой структуры с триггерами для слияния
- **Адаптивность**: выбор стратегии на основе:
  - Размера данных
  - Паттернов доступа (read-heavy vs write-heavy)
  - Соотношения операций чтения и записи
- **Пороговые значения**: настраиваемые параметры для запуска compaction

### 7. Range Scan

Реализуйте операцию `scan(from_key, to_key)`:

- **Поиск по диапазону**: возврат всех записей в заданном диапазоне ключей
- **Multi-file scan**: чтение из нескольких SSTable файлов одновременно
- **Merge-сортировка**: объединение результатов из MemTable и SSTable
- **Итератор**: реализация ленивого итератора для экономии памяти
- **Обработка tombstones**: корректное игнорирование удаленных записей

### 8. Bloom Filter

Оптимизируйте операции чтения с помощью фильтров Блума:

- **Bloom Filter для каждой SSTable**: вероятностная структура данных для быстрой проверки отсутствия ключа
- **False positive rate**: настраиваемый уровень ложноположительных срабатываний (рекомендуется 1-5%)
- **Сериализация**: сохранение Bloom Filter вместе с SSTable
- **Пропуск файлов**: использование Bloom Filter для исключения файлов без нужного ключа

## Основные операции

### PUT (запись)
1. Запись в WAL для надежности
2. Вставка в MemTable
3. Если MemTable переполнена → flush в SSTable L0
4. Удаление старого WAL

### GET (чтение)
1. Поиск в MemTable
2. Если не найдено → поиск в SSTable (от новых к старым)
3. Использование Bloom Filter для пропуска файлов
4. Memory-mapped access для эффективного чтения

### DELETE (удаление)
1. Запись tombstone в MemTable
2. Физическое удаление во время compaction

### SCAN (диапазонный поиск)
1. Сбор итераторов из MemTable и всех SSTable
2. K-way merge с учетом tombstones
3. Возврат отсортированных результатов в диапазоне

### COMPACTION (слияние)
1. Выбор файлов для слияния согласно политике
2. K-way merge с удалением tombstones и дубликатов
3. Запись результата в новую SSTable
4. Удаление старых файлов

## Требования к производительности

- **PUT**: O(log N) в памяти + O(1) для WAL
- **GET**: O(log N) для MemTable + O(K * log M) для SSTable с Bloom Filter
- **SCAN**: O(K * M) где K - количество файлов, M - размер диапазона
- **Compaction**: должен работать в фоновом режиме без блокировки операций чтения/записи

## Дополнительные требования

1. **Потокобезопасность**: все операции должны быть thread-safe
2. **Crash recovery**: восстановление состояния после сбоя из WAL
3. **Настраиваемость**: параметры (размер MemTable, политика compaction) через конфигурацию
4. **Метрики**: сбор статистики (количество операций, задержки, размер данных)
5. **Тестирование**: юнит-тесты и интеграционные тесты для всех компонентов
